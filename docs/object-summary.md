# Сводки объектов (Object Summary)

Отдельный слой поверх графа: для объектов метаданных LLM генерирует связную **сводку назначения**, по
которой затем можно искать. Это самостоятельный конвейер генерации и эмбеддинга со своими настройками,
поэтому вынесен в отдельный документ. Публикуется как инструмент `find_objects_by_summary`.

## Зачем это нужно

Метаданные 1С описывают объект структурно (реквизиты, типы, связи), но не отвечают на вопрос «зачем
этот объект и что он делает в прикладном смысле». Сводка — это сгенерированное человекочитаемое
описание роли объекта в конфигурации. Поиск по сводкам находит объекты по функциональному смыслу,
даже когда структурные признаки и имена не помогают.

## Включение

- `OBJECT_SUMMARY_ENABLED` — главный флаг. По умолчанию выключен; при `false` поведение проекта не
  меняется и инструмент `find_objects_by_summary` не публикуется.
- `OBJECT_SUMMARY_GENERATION_MODE`:
  - `auto` — сводки генерируются в фоне при старте;
  - `manual` — при старте выполняется только сверка (S0), а генерация запускается вручную для каждого
    объекта из веб-консоли.
- `OBJECT_SUMMARY_CATEGORIES` — для каких категорий объектов строить сводки (подмножество поддерживаемых
  категорий; формат — JSON-массив).
- `OBJECT_SUMMARY_PROFILE_SIZE_POLICY` — объём профиля объекта, отправляемого в LLM: `small` / `medium`
  / `large`. Больше профиль — богаче контекст и дороже генерация.

## Как устроен конвейер: фазы S0 → S1 → S2

Генерация сводок разбита на три фазы:

- **S0 — сверка (reconcile).** Определяет, для каких объектов сводка отсутствует, устарела или должна
  быть перегенерирована (с учётом версий профиля). Выполняется всегда, в том числе в режиме `manual`.
- **S1 — генерация (только `auto` или ручной запуск).** Для объекта собирается профиль (структурные
  факты и, при необходимости, фрагменты кода), кодируется в компактный формат (TOON) для экономии
  токенов, отправляется в LLM, и из ответа рендерится сводка.
- **S2 — эмбеддинг.** Для объектов, у которых есть сводка, но нет векторного представления, строится
  документ эмбеддинга и генерируется вектор для семантического поиска.

Фазы независимы: сводка может быть уже сгенерирована (S1), но ещё не проэмбеддена (S2) — тогда она
видна в консоли, но пока не находится семантическим поиском.

## LLM-канал

Сводки используют **отдельный** LLM-канал (`OBJECT_SUMMARY_LLM_API_BASE`, `OBJECT_SUMMARY_LLM_API_KEY`,
`OBJECT_SUMMARY_MODEL`, `OBJECT_SUMMARY_LLM_TEMPERATURE`, таймаут, прокси). 

Эмбеддинг сводок использует общие настройки эмбеддингов (`EMBEDDING_*`), а параллелизм этой фазы —
свои `OBJECT_SUMMARY_EMBEDDING_WORKERS` / `OBJECT_SUMMARY_EMBEDDING_BATCH_SIZE`. Параллелизм и повторы
генерации настраиваются группой `OBJECT_SUMMARY_GENERATION_*`.

### Выбор модели

По результатам экспериментов оптимальной по соотношению «цена/качество» оказалась модель
**`gpt-5.4-mini`** — она и стоит по умолчанию в `OBJECT_SUMMARY_MODEL`.

- **Более слабые модели** опирались не только на факты из профиля объекта, но и додумывали недостающее
  в пределах доменных знаний (общих представлений о том, как «обычно» устроены такие объекты в 1С).
  Итоговая сводка выглядела правдоподобно, но не всегда соответствовала фактической функциональности
  конкретного объекта.
- **Более дорогие модели** заметно лучшего результата не давали: прирост качества сводок не оправдывал
  роста стоимости генерации.

## Где хранятся сводки

Сводки хранятся **файлами** в каталоге `data/object_summary/` — по отдельной папке на объект (путь
вида `конфигурация/категория/объект`). В папке объекта:

- `summary.json` — сама сводка, источник истины;
- `summary.md` — та же сводка в Markdown (для отображения);
- `profile.toon` — профиль объекта, который отправлялся в LLM;
- `meta.json` — версии и метки, по которым S0-сверка решает, нужна ли перегенерация.

В **Neo4j сама сводка целиком не хранится** — там лежат только путь к `summary.json`, эмбеддинг и текст
для поиска (плюс векторное представление в векторном индексе). Поэтому сводки видны в поиске и в
карточке объекта, но их содержимое остаётся в файлах.

## Ручная генерация из веб-консоли

В режиме `manual` (и при необходимости в `auto`) сводки можно создавать и обновлять из веб-консоли:

- кнопка **«Создать»** доступна для объектов без сводки;
- кнопка **«Обновить»** (перегенерация существующей) управляется флагом
  `OBJECT_SUMMARY_MANUAL_REGENERATION_ENABLED`: при `false` обновлять уже существующие сводки вручную
  нельзя.

Подробнее об интерфейсе — в [web-console.md](web-console.md).

## Поиск по сводкам

Инструмент `find_objects_by_summary` ищет объекты по их сводкам семантически. Поверх результата можно
включить переранжирование (`OBJECT_SUMMARY_RERANK_ENABLED`, `OBJECT_SUMMARY_RERANK_TOP_K`; модель —
общие `RERANK_*`). Как режим поиска сводки кратко упомянуты в [search.md](search.md).

## Сводки для расширений

Генерацию сводок можно распространить на объекты расширений: `OBJECT_SUMMARY_GENERATE_FOR_EXTENSIONS`,
а также ограничить конкретными расширениями (`OBJECT_SUMMARY_EXTENSION_NAMES`) и областью объектов
(`OBJECT_SUMMARY_EXTENSION_OBJECT_SCOPE`). Про сами расширения — [extensions.md](extensions.md).

## Версионирование и перегенерация

При изменении версии профиля перегенерация существующих сводок по умолчанию **не** запускается
автоматически — это управляется `OBJECT_SUMMARY_REGENERATE_ON_PROFILE_VERSION_CHANGE`. Так смена
формата профиля не приводит к незапланированной массовой (и платной) перегенерации.

## Типовые проблемы

- **`find_objects_by_summary` отсутствует** — не включён `OBJECT_SUMMARY_ENABLED` (или инструмент
  скрыт в веб-консоли).
- **Сводки не генерируются автоматически** — режим `manual`; запускайте генерацию из консоли, либо
  переключите на `auto`.
- **Сводка есть, но поиск её не находит** — не прошла фаза S2 (эмбеддинг) или не настроены `EMBEDDING_*`.
- **Генерация падает / «degraded»** — как правило, недоступен LLM- или embedding-endpoint; проверьте
  `OBJECT_SUMMARY_LLM_*` и `EMBEDDING_*`.
