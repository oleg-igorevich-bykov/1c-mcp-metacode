# Поиск

Поиск — основной способ работы с системой. Режимов несколько, и они закрывают разные задачи. Этот
документ — навигатор: какой режим для чего и как он устроен. Два самых объёмных режима вынесены в
отдельные документы: поиск по телу кода — [bsl-code-search.md](bsl-code-search.md), поиск по сводкам
объектов — [object-summary.md](object-summary.md).

## Какой режим выбрать

| Задача | Режим / инструмент |
|--------|--------------------|
| Найти объект по точным структурным признакам (тип, реквизиты, связи) | структурный поиск |
| Найти объект «по смыслу», когда точного имени нет | поиск по описанию |
| Найти реквизит/ресурс/измерение/команду внутри объектов | `find_metadata_elements` |
| Понять, где объект используется | `find_metadata_usages` |
| Найти процедуру/функцию по имени, сигнатуре или описанию | `search_bsl_routines` |
| Найти код по тому, что он делает (по телу) | `search_bsl_code` → [bsl-code-search.md](bsl-code-search.md) |
| Найти объект по развёрнутой сводке его назначения | `find_objects_by_summary` → [object-summary.md](object-summary.md) |

Общий принцип: **большую часть информации можно найти структурным поиском, без LLM** — задав нужные
признаки явными параметрами инструмента (тип объекта, имя, состав, связи). Семантические режимы (по
описанию, по коду, по сводке) нужны, когда запрос сформулирован на естественном языке и точных
признаков нет.

## Структурный поиск

Поиск по свойствам и связям графа: тип объекта, состав реквизитов, участие в движениях, права и т.д.
Не требует эмбеддингов и работает всегда.

- `find_metadata_objects` в структурном режиме — объекты по признакам;
- `find_metadata_elements` — элементы (реквизиты, ресурсы, измерения, команды) внутри объектов;
- `find_metadata_usages` — обратные связи: где объект задействован.

Это самый точный и предсказуемый способ, когда известно, что именно ищется.

## Поиск по описанию

`find_metadata_objects` в режиме поиска по описанию ищет объекты по их смысловому содержанию:
описаниям, комментариям, синонимам, справке и внутренним именам. Поддерживает три подрежима:

- **fulltext** — точное совпадение по ключевым словам (полнотекстовый индекс);
- **vector** — семантическая близость по эмбеддингам (требует включённой векторной индексации
  описаний метаданных);
- **hybrid** — комбинация двух веток.

Поиск процедур/функций по описанию работает аналогично через `search_bsl_routines` (режим по
описанию), поверх описаний, извлечённых из комментариев над сигнатурами.

## Гибридный поиск: как он устроен

Гибридный режим объединяет полнотекстовую и векторную ветки в один ранжированный список. Ключевые
принципы (все параметры — в секции гибридного поиска `.env.example`):

- **Веса веток** — вклад fulltext и vector в итоговый скор (`HYBRID_SEARCH_FULLTEXT_WEIGHT`,
  `HYBRID_SEARCH_VECTOR_WEIGHT`). Рекомендуемый баланс смещён в сторону векторной ветки.
- **Нормализация** (`HYBRID_NORMALIZATION_STRATEGY`) — приводит скоры двух веток к общей шкале перед
  слиянием; перцентильная нормализация снижает влияние выбросов.
- **Слияние** (`HYBRID_FUSION_MODE`) — взвешенная сумма или RRF (Reciprocal Rank Fusion). RRF не
  зависит от абсолютной шкалы скоров и служит робастным базовым вариантом.
- **Динамические веса** (`HYBRID_DYNAMIC_WEIGHTS_ENABLED`) — адаптация баланса под конкретный запрос:
  короткие запросы тяготеют к точному совпадению, длинные — к семантике.
- **Устойчивая пагинация** — перевыбор кандидатов (`HYBRID_OVERSAMPLE_FACTOR`) и ограничение глубины
  извлечения из каждой ветки (`HYBRID_EFF_K_CAP`) для стабильности результата и латентности.

## Rerank

Поверх результатов поиска по описанию можно включить переранжирование внешней rerank-моделью — она
пересортировывает верхушку выдачи по релевантности запросу. Настройки самой модели общие
(`RERANK_*`), включение — своё для каждого вида поиска:

- `METADATA_DESCRIPTION_RERANK_ENABLED` — для поиска метаданных по описанию;
- `ROUTINE_DESCRIPTION_RERANK_ENABLED` — для поиска процедур по описанию;
- `OBJECT_SUMMARY_RERANK_ENABLED` — для поиска по сводкам;
- `BSL_CODE_RERANK_ENABLED` — для поиска по коду.

У каждого — свой `*_RERANK_TOP_K`: сколько верхних кандидатов отдавать на переранжирование.

## Типовые проблемы

- **Векторный/гибридный поиск ничего не находит или ведёт себя как fulltext** — не включена или ещё
  не завершилась векторная индексация описаний (`ENABLE_METADATA_DESCRIPTION_EMBEDDING` /
  `ENABLE_ROUTINE_DESCRIPTION_EMBEDDING`). Индексация идёт в фоне после загрузки данных.
- **Поиск по коду недоступен** — это отдельная подсистема, см. [bsl-code-search.md](bsl-code-search.md).
- **Rerank не срабатывает** — проверьте, что заданы `RERANK_*` и включён нужный `*_RERANK_ENABLED`.
