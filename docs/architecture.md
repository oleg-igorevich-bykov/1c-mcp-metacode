# Архитектура и модель данных

Документ о том, **как устроен проект внутри**: из каких частей он состоит, как данные проходят
путь от выгрузки 1С до ответа MCP-клиенту, что именно хранится в графовой базе, а что — в отдельных
SQLite-файлах. Читайте, если хотите понять картину целиком; для запуска достаточно [setup.md](setup.md).

## Из чего состоит система

Система собирается из четырёх слоёв, работающих в одном контейнере приложения (плюс отдельный
контейнер Neo4j):

1. **Загрузчик** — читает выгрузку конфигурации 1С (отчёт по конфигурации и/или XML-дамп) и строит
   граф в Neo4j: объекты метаданных, формы, модули, процедуры, права, предопределённые значения,
   подписки на события, расширения.
2. **Графовая база Neo4j** — центральное хранилище модели. Всё, что можно выразить как объекты и
   связи между ними, живёт здесь.
3. **MCP-сервер** — публикует инструменты (tools) поверх графа: структурный поиск, поиск по описанию,
   получение структуры объектов и форм, работа с кодом BSL. Это API-контракт для внешних агентов и IDE.
4. **Веб-консоль и встроенный агент** — UI поверх того же графа: дерево метаданных, карточки объектов,
   просмотр кода, статистика, управление видимостью tools, а также AI агент, который ходит в те же
   инструменты, что и внешние клиенты.

Поток данных линейный: **выгрузка 1С → загрузчик → Neo4j → (MCP-сервер | веб-консоль | агент) →
пользователь**. Загрузчик работает при старте и затем поддерживает граф в актуальном состоянии
инкрементально (см. [loading-and-updates.md](loading-and-updates.md)).

## Модель графа

Полная диаграмма модели — в [README](../README.md#структура-данных). Здесь — словарь: какие бывают
узлы и какие связи их соединяют.

### Основные узлы

- **Project** → **Configuration** — корень. **Ключевой момент концепции: проект содержит не одну, а
  несколько конфигураций** (связь `HAS_CONFIGURATION`). Проект в этом приложении — это базовая
  конфигурация 1С **вместе со всеми её расширениями**: базовая конфигурация и каждое расширение
  представлены отдельными узлами `Configuration` под общим `Project`. Расширения помечены признаком
  `is_extension`, и конфигурация расширения связана с базовой конфигурацией связью
  `(расширение)-[:EXTENDS]->(базовая)`. Именно поэтому базовая конфигурация и её расширения живут в
  одном графе одного проекта и имеют между собой связи — см. раздел «Связи расширений» ниже и
  [extensions.md](extensions.md).
- **MetadataCategory** — категория метаданных (Справочники, Документы, Регистры и т.д.);
  конфигурация связана с категориями через `HAS_CATEGORY`.
- **MetadataObject** — объект метаданных (конкретный справочник, документ, регистр…). Центральный
  узел модели, к нему привязано большинство остальных сущностей.
- **Attribute**, **TabularPart**, **Resource**, **Dimension** — реквизиты, табличные части, ресурсы
  и измерения объектов и регистров.
- **Form** → **FormControl** / **FormEvent** / **FormAttribute** / **FormEventAction** — структура
  управляемых форм: элементы, события, реквизиты и действия событий. Для обычных форм структура не
  загружается — от них берётся только код модуля формы (см. Module/Routine ниже).
- **Module** → **Routine** → **RoutineCodeUnit** — модули, объявленные в них процедуры/функции и
  единицы кода процедур (для поиска по телу кода). Сюда попадает и код модуля формы обычных форм.
- **Command**, **UrlTemplate** → **UrlMethod** — команды и HTTP-сервисы с их методами.
- **PredefinedItem** — предопределённые значения (с иерархией через `HAS_CHILD`).
- **EnumValue**, **Characteristic**, **AccountingFlag**, **DimensionAccountingFlag**, **Layout**,
  **JournalGraph** — специализированные дочерние сущности объектов (значения перечислений, признаки
  учёта планов счетов, макеты, графы журналов расчёта).
- **Role** — роли; права выражаются связью `GRANTS_ACCESS_TO` к целевым объектам.

### Основные связи

| Связь | Смысл |
|-------|-------|
| `HAS_CONFIGURATION`, `HAS_CATEGORY` | иерархия проект → конфигурация → категория |
| `CONTAINS_OBJECT` | вхождение объекта (в т.ч. в подсистемы) |
| `HAS_ATTRIBUTE`, `HAS_TABULAR_PART`, `HAS_RESOURCE`, `HAS_DIMENSION` | состав объекта/регистра |
| `HAS_FORM`, `HAS_CONTROL`, `HAS_EVENT`, `HAS_FORM_ATTRIBUTE`, `HAS_EVENT_ACTION` | формы и их содержимое |
| `HAS_MODULE`, `DECLARES`, `CALLS`, `HAS_CODE_UNIT`, `OF_ROUTINE` | модули, процедуры, граф вызовов, единицы кода |
| `HAS_HANDLER`, `USES_HANDLER` | привязка событий/команд/подписок к процедурам-обработчикам |
| `USED_IN` | использование объекта в реквизитах, элементах форм и т.д. |
| `DO_MOVEMENTS_IN` | движения документа по регистрам |
| `GRANTS_ACCESS_TO` | права роли на объект |
| `HAS_COMMAND`, `LINKS_TO_COMMAND`, `BINDS_TO` | команды и связывание элементов форм |
| `HAS_URL_TEMPLATE`, `HAS_URL_METHOD` | HTTP-сервисы |
| `HAS_PREDEFINED`, `HAS_ENUM_VALUE`, `HAS_CHARACTERISTIC`, `HAS_LAYOUT`, `HAS_GRAPH`, `HAS_ACCOUNTING_FLAG`, `HAS_DIMENSION_ACCOUNTING_FLAG` | специализированные дочерние сущности |
| `HAS_EVENT_SUBSCRIPTION` | подписки на события |

### Связи расширений

Объекты расширения хранятся в том же графе, что и базовая конфигурация, и связаны с ней отдельными
типами связей на разных уровнях:

- `EXTENDS` — **на уровне конфигураций**: конфигурация расширения указывает на базовую конфигурацию
  (`Configuration → Configuration`). Это связь, которая держит базу и её расширения в одном проекте.
- `ADOPTED_FROM` — объект расширения заимствован из базового (адаптированный/заимствованный объект,
  форма, элемент формы, событие указывают на свой базовый оригинал);
- `EXTENDS_MODULE`, `EXTENDS_ROUTINE`, `EXTENDS_ACTION` — на уровне содержимого: расширение
  дополняет/переопределяет базовый модуль, процедуру или действие события.

Подробнее о представлении расширений — в [extensions.md](extensions.md).

## Где что хранится

Основная модель живёт в **Neo4j**. Помимо графа система использует несколько независимых хранилищ —
SQLite-файлы-сайдкары и файловый каталог сводок — каждое для своей задачи. Разделение сделано
намеренно: это разные жизненные циклы данных, и держать их в графе было бы неудобно.

| Хранилище | Что содержит | Настройка пути |
|-----------|--------------|----------------|
| **Neo4j** | вся модель графа: метаданные, формы, модули, процедуры, права, предопределённые значения, подписки, расширения, а также векторные индексы описаний и кода | `NEO4J_*` |
| **SQLite: incremental state** | состояние инкрементальной загрузки (что уже загружено, watermark-и, блокировка планировщика) | `INCREMENTAL_LOADING_STATE_PATH` |
| **SQLite: BSL code search** | лексический индекс тел процедур для поиска по коду (BM25/RLM, полевые оценки) | `BSL_CODE_SEARCH_SQLITE_PATH` |
| **Файлы: object summary** | сгенерированные сводки по объектам (`summary.json/.md`, профиль, метки версий) — по папке на объект; в Neo4j только путь, эмбеддинг и текст для поиска | каталог `data/object_summary/` |
| **SQLite: веб-консоль** | пользователи консоли и их токены | `WEB_CONSOLE_USERS_SQLITE_PATH` |
| **SQLite: MCP runtime** | видимость MCP-tools и рантайм-настройки | `MCP_SETTINGS_SQLITE_PATH` |
| **SQLite: агент** | сессии и чаты встроенного агента | `CONSOLE_AGENT_SESSION_SQLITE_PATH`, `CONSOLE_AGENT_CHATS_SQLITE_PATH` |

Важная деталь развёртывания: SQLite-сайдкары по умолчанию лежат в каталоге `storage/` внутри
контейнера, который монтируется на именованный volume (а **не** в bind-mounted `data/`). Это отделяет
служебное состояние от входных данных 1С: пересоздание `data/` не сбрасывает состояние загрузки, а
пересоздание volume со `storage/` заставляет систему переиндексировать всё с нуля.

## Конфигурация

Все настройки задаются переменными окружения (файл `.env`) и/или в `docker-compose.yml` для значений,
различающихся между проектами. **Полный перечень переменных с дефолтами и комментариями — в файле
`.env.example`**; это единственный актуальный реестр настроек. Тематические документы описывают
*поведение* ключевых флагов, но не перечисляют переменные целиком.

## Мультипроектность

В одном развёртывании может работать несколько проектов (конфигураций) — по одному контейнеру
приложения на проект, с общей или раздельной базой Neo4j. При поиске указывать проект не нужно:
сессия знает свой `PROJECT_NAME` и связанные проекты (`RELATED_PROJECTS`) и фильтрует автоматически.
