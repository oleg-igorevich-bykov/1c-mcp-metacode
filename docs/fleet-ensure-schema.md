# Fleet: предсоздание схемы Neo4j (`--ensure-schema` + `SCHEMA_MANAGED_EXTERNALLY`)

Инструкция для разработчика инфраструктуры флота (`mcp-fleet` / `infra-mcp-fleet`).

## Зачем

Граф-схема (constraints, indexes, fulltext) в 1c-mcp-metacode — **глобальная и общая**
на весь флот: один инстанс Neo4j Community, одна база `neo4j`, метки (`:Module`,
`:Routine`, `:MetadataObject`, …) не разделены по проектам.

Когда `apply-fleet` поднимает 20+ проектных контейнеров одновременно, каждый при
первичной загрузке метаданных вызывает `create_indexes()` и все они **дерутся за
schema-локи**:

```
WARNING - Could not create constraint/index:
{neo4j_code: Neo.TransientError.Transaction.DeadlockDetected}
{message: ForsetiClient[...] can't acquire UpdateLock{...} on LABEL(4) ...}
```

Изменение схемы берёт широкий лок на метку — это худший вид контеншена и главный
тормоз первичной индексации флота. Ретраи в приложении это лишь смягчают.

Решение: создать схему **один раз, одним писателем, ДО** старта проектных контейнеров,
а контейнерам сказать не трогать DDL. Тогда при параллельной загрузке остаются только
гранулярные data-write локи, которые Neo4j разруливает нормально.

## Что появилось в приложении

- **CLI-режим `python main.py --ensure-schema`** — идемпотентно создаёт всю схему
  (constraints/indexes/fulltext + vector, если embedding-endpoint доступен) как один
  писатель и завершается с кодом `0`/`1`. MCP-сервер не поднимает, метаданные не грузит.
  Всё DDL идёт через `IF NOT EXISTS` — запускать можно сколько угодно раз.
- **Env-флаг `SCHEMA_MANAGED_EXTERNALLY`** (по умолчанию `false`). При `true` контейнер
  пропускает `create_indexes()` при загрузке метаданных. Векторные индексы всё равно
  досоздаются per-container (они зависят от embedding-endpoint, который на момент
  ensure-schema мог быть недоступен) — это безопасная страховка.

## Что нужно сделать во флоте

### 1. Добавить шаг ensure-schema перед стартом проектных контейнеров

В пайплайн `apply-fleet` (перед `docker compose up -d` проектных сервисов) добавить
one-shot запуск на **том же** образе, в **той же** docker-сети, с **тем же** `.env`
(нужны только `NEO4J_URI` / `NEO4J_PASSWORD` — project-специфичные переменные не важны):

```bash
docker run --rm \
  --network <fleet-net> \
  --env-file <path>/.env \
  <mcp-metacode-image> \
  python main.py --ensure-schema
```

- Дождаться **успешного завершения** (exit code `0`) до старта проектных контейнеров.
  При `1` — не поднимать флот: Neo4j недоступен или DDL не прошёл, разберитесь с этим
  сначала (иначе контейнеры с `SCHEMA_MANAGED_EXTERNALLY=true` стартуют без схемы).
- Шаг идемпотентный — можно (и стоит) вызывать перед **каждым** apply, а не только на
  первый разворот. Стоит дёшево, гарантирует, что новые индексы из свежих версий кода
  доедут до общей БД до старта контейнеров.
- Желательно, чтобы embedding-endpoint был **поднят до** этого шага — тогда ensure-schema
  создаст и векторные индексы сразу. Если нет — не страшно, их досоздаст каждый контейнер.

### 2. Выставить `SCHEMA_MANAGED_EXTERNALLY=true` проектным контейнерам

Только после того, как шаг (1) встроен в пайплайн. В `.env`/compose проектных сервисов:

```
SCHEMA_MANAGED_EXTERNALLY=true
```

Без шага (1) этот флаг ставить **нельзя** — контейнеры пропустят создание схемы, а
создавать её будет некому.

### 3. (Опционально, но рекомендуется) поднять параллелизм bootstrap

Раньше приходилось душить одновременный старт (`BOOTSTRAP_BATCH_SIZE`), потому что
контейнеры дедлочились на схеме. После выноса схемы этот класс дедлоков уходит, и батчи
можно **увеличить** — параллелить загрузку теперь безопаснее. Подбирайте по факту,
наблюдая метрики (см. `/api/console/metrics/index`) и лог Neo4j.

## Как проверить

1. **Прямой прогон ensure-schema:**
   ```bash
   docker run --rm --network <net> --env-file .env <image> python main.py --ensure-schema
   echo "exit=$?"     # ожидаем 0
   ```
   В логе — `Ensuring Neo4j schema ...` → `Neo4j schema ensured.`

2. **Схема на месте (в Neo4j Browser / cypher-shell):**
   ```
   SHOW CONSTRAINTS;   // должны присутствовать все 26 ожидаемых uq_*
   SHOW INDEXES;
   ```

3. **Контейнеры при массовом старте больше не дедлочатся:** в логах проектных
   контейнеров с `SCHEMA_MANAGED_EXTERNALLY=true` при первичной загрузке **нет**
   `Could not create constraint/index` / `DeadlockDetected` на CREATE CONSTRAINT/INDEX.

## Порядок отката

Полностью обратимо: убрать `SCHEMA_MANAGED_EXTERNALLY` (или `=false`) у контейнеров —
они снова начнут создавать схему сами, как раньше. Шаг ensure-schema можно оставить в
пайплайне: он идемпотентен и с `SCHEMA_MANAGED_EXTERNALLY=false` просто безвреден
(контейнер повторно прогонит те же `IF NOT EXISTS`).

## Что это НЕ решает

Убирает schema-лок контеншен (главный источник дедлоков при массовом bootstrap). Data-write
контеншен от 20+ писателей в один инстанс остаётся — но он гранулярный и переживается через
managed-транзакции/ретраи. Если после этого шага скорость всё ещё упирается в запись,
следующие рычаги — джиттер старта (`STARTUP_DELAY_SECONDS`), размер пула соединений на
контейнер (`NEO4J_POOL_SIZE`), и уже архитектурные варианты (Neo4j Enterprise multi-DB /
стор-на-контейнер).
