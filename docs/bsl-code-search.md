# Поиск по коду BSL

Семантический поиск по **телу** процедур и функций 1С — отдельная подсистема, а не «ещё один режим
поиска». Она строит собственный индекс, работает в две фазы и имеет свои настройки. Публикуется как
инструмент `search_bsl_code`. Поиск процедур по имени/сигнатуре/описанию — это другое, см.
[search.md](search.md).

## Зачем это нужно

Обычный поиск по описанию находит процедуры по комментариям над сигнатурой. Поиск по коду находит
процедуры по тому, **что реально написано в теле** — по логике, вызовам, работе с объектами, — даже
если описания нет. Это полезно, когда нужно найти реализацию по смыслу («где считается себестоимость»,
«где формируется движение по регистру»).

## Включение

- `ENABLE_BSL_CODE_SEARCH` — главный флаг. Включает индексатор кода, sidecar-хранилище индекса и сам
  инструмент `search_bsl_code`. Без него инструмент не публикуется.
- `ENABLE_BSL_CODE_EMBEDDING` — включает векторную (семантическую) ветку. При `false` индекс всё равно
  строится, но поиск идёт только через лексический механизм (RLM), без эмбеддингов.

Итоговое правило: векторная ветка активна, только когда включены **оба** флага. Лексическая ветка
работает при включённом `ENABLE_BSL_CODE_SEARCH` всегда.

Подсистема работает поверх тел процедур, которые загружает загрузчик сигнатур BSL, поэтому требует
также включённого `LOAD_BSL_SIGNATURES` (по умолчанию включён). Если его выключить, инструмент
`search_bsl_code` не публикуется даже при `ENABLE_BSL_CODE_SEARCH=true`.

## Как устроен индекс: две фазы

Индексация тел процедур разбита на две независимые по нагрузке фазы:

1. **Фаза A — подготовка (нагрузка на CPU).** Тела процедур нарезаются на юниты с учётом синтаксиса
   (подробнее см. [Как код нарезается на юниты](#как-код-нарезается-на-юниты)), токенизируются, из них
   извлекаются структурные признаки. Результат складывается в лексический индекс (sidecar SQLite с
   полнотекстовым механизмом для BM25/RLM и полевых оценок).
2. **Фаза B — эмбеддинги (нагрузка на сеть/IO).** Для юнитов генерируются векторные представления
   через embedding-API и записываются в векторный индекс Neo4j (при необходимости текст перед этим
   сжимается — см. [Компрессия текста для эмбеддинга](#компрессия-текста-для-эмбеддинга)). Эта фаза
   выполняется, только если включён `ENABLE_BSL_CODE_EMBEDDING`.

Разделение на фазы позволяет параллелить CPU-работу и сетевые вызовы независимо и переживать перезапуски
(индексация возобновляется с места остановки).

При первичной индексации (при старте приложения) фазы могут идти параллельно: как только Фаза A
фиксирует первые фрагменты, Фаза B начинает эмбеддить их, не дожидаясь завершения всей Фазы A. На
больших конфигурациях это заметно сокращает время до готового семантического поиска. Управляется флагом
`BSL_CODE_STARTUP_PHASE_B_OVERLAP_ENABLED` (по умолчанию включён); плановая и инкрементальная индексация
работают как прежде, последовательно.

Параллельность не ослабляет гарантий: Фаза A не зависит от доступности embedding-API — если он недоступен,
лексический индекс всё равно достраивается и поиск работает по словам (RLM), а векторная ветка догоняется
позже (повторными попытками или на следующем цикле). Пока индекс не готов полностью, семантическая ветка
им не пользуется. При старте система дольше ждёт восстановления недоступного embedding-API, чем в плановых
циклах, — на случай, если он поднимется уже после длинной Фазы A.

## Как код нарезается на юниты

Тело процедуры перед индексацией превращается в один или несколько **юнитов** — минимальных
единиц, которые попадают в индекс и возвращаются в выдаче. Как именно режется тело, задаёт
`BSL_CODE_SPLIT_STRATEGY`. Работают два уровня логики:

- **Короткая процедура — один юнит.** Если тело помещается в окно нарезки, вся процедура целиком
  становится одним юнитом, без разбиения.
- **Длинная процедура — «ast-safe» скользящее окно.** Тело режется окном с перекрытием, но границы
  ставятся не «вслепую» по счётчику символов, а притягиваются к безопасным точкам синтаксиса,
  которые находит парсер BSL (tree-sitter). Юнит не обрывается посреди оператора и не заканчивается
  на строке, которая только открывает блок (`Если … Тогда`, `Для … Цикл`, `Попытка`). Перекрытие
  соседних юнитов не даёт потеряться логике на стыке окон.
- **Уточнение по языку запросов.** Внутри строковых литералов с запросами (`ВЫБРАТЬ …`) границы
  дополнительно выравниваются по секциям запроса (`ИЗ`, `ГДЕ`, `СГРУППИРОВАТЬ` и т.п.). Это делается
  по возможности: если разбор запроса недоступен, нарезка обходится границами BSL.

Доступны два режима нарезки; отличаются размером окна и перекрытия:

- `ast_safe_sliding_3600_720_min480` (по умолчанию) — **крупные юниты**: окно ~3600 символов,
  перекрытие ~720. Юнитов меньше, в каждом больше контекста; индексация быстрее и эмбеддингов
  меньше, но попадание в поиске чуть грубее.
- `ast_safe_sliding_2200_440_min300` — **мелкие юниты**: окно ~2200 символов, перекрытие ~440.
  Точнее попадание в нужный фрагмент, но юнитов и эмбеддингов больше.

Парсер BSL обязателен для нарезки длинных процедур: без него длинное тело не режется и его
индексация завершается ошибкой (короткие процедуры, укладывающиеся в одно окно, не затрагиваются).

## Компрессия текста для эмбеддинга

Компрессия влияет **только на текст, который уходит в embedding-модель** (Фаза B). Лексический поиск
(RLM), полевые оценки и выдача фрагментов всегда работают по сырому тексту процедуры — компрессия их
не меняет и в результатах поиска не видна. Смысл — убрать из текста структурный шум (ключевые слова
языка, повторы идентификаторов), чтобы вектор концентрировался на смысловых токенах. Режим задаёт
`BSL_CODE_COMPRESSION_STRATEGY`:

- `none` (по умолчанию) — компрессии нет: в эмбеддинг уходит сырой текст юнита с добавленным
  контекстом-заголовком (объект / форма / процедура).
- `lexdedup_terms_cap1_lines_normprefix` — **лексический дедуп, только идентификаторы**: выбрасываются
  ключевые слова языка, каждый идентификатор оставляется один раз, а цепочки вида `Объект.Свойство`
  разбираются на отдельные токены.
- `lexdedup_cap1_nochainparts_lines_normprefix` — **лексический дедуп с цепочками**: то же, но цепочки
  `Объект.Свойство` сохраняются целиком (а их части не дублируются отдельными токенами). Полезно,
  когда важны обращения к реквизитам и методам объектов.
- `rawbelow1000_lexdedup_terms_cap1_lines_normprefix` — **порог raw<1000 поверх дедупа идентификаторов**:
  короткие юниты (сырое тело меньше 1000 символов) уходят в эмбеддинг сырыми, как при `none`; длинные
  сжимаются как `lexdedup_terms_cap1_lines_normprefix`. Экономит символы в основном на крупных юнитах,
  сохраняя точность на коротких.
- `rawbelow1000_lexdedup_cap1_nochainparts_lines_normprefix` — **порог raw<1000 поверх дедупа с
  цепочками**: то же правило порога, но длинные юниты сжимаются как
  `lexdedup_cap1_nochainparts_lines_normprefix` (цепочки целиком).

В режимах дедупа перед сжатым текстом ставится нормализованный заголовок-комментарий с объектом,
формой и именем процедуры; у пороговых режимов короткие юниты, оставленные сырыми, получают тот же
заголовок, что и при `none`. Компрессия никогда не делает текст длиннее исходного: если сжатый вариант
получился не короче, берётся исходный.

Обе настройки — нарезка и компрессия — входят в отпечаток конфигурации индекса, поэтому смена любой из
них приводит к полной переиндексации (см. [Переиндексация](#переиндексация)).

Сравнение режимов нарезки, компрессии и retrieval по результатам замеров (по которым выбирались рабочие
значения) — в [bsl-code-search-benchmark.md](bsl-code-search-benchmark.md).

## Как выполняется поиск

- **Векторная ветка** (если включена) ищет по смысловой близости эмбеддингов; из Neo4j берётся пул
  кандидатов, размер которого настраивается.
- **RLM (лексическая ветка)** ранжирует по совпадению терминов и полевым оценкам поверх
  полнотекстового индекса. Служит как fallback при выключенных эмбеддингах и как отдельный механизм
  ранжирования.
- **Rerank** (`BSL_CODE_RERANK_ENABLED`) — опциональное переранжирование верхушки внешней моделью
  (`BSL_CODE_RERANK_TOP_K` кандидатов; сама модель настраивается общими `RERANK_*`).
- **Пагинация курсором** — результаты отдаются порциями, устойчиво к изменению корпуса между
  запросами.

Лимит результатов по умолчанию — `BSL_CODE_SEARCH_DEFAULT_LIMIT` (обычно ниже общего лимита запросов
ради скорости).

## Что попадает в поиск: coverage policy

Не весь код обязательно индексируется и виден в поиске по умолчанию:

- `BSL_CODE_EMBEDDING_EXCLUDED_OWNER_CATEGORIES` — список категорий-владельцев (например, отдельные
  виды объектов), которые не эмбеддятся и скрыты в поиске по умолчанию. Чтобы искать в них, клиент
  должен явно указать эти категории в запросе. Пустой список — не исключать ничего.

Это сделано, чтобы крупные, но редко нужные для поиска по коду категории не раздували индекс и не
зашумляли выдачу.

## Хранилище индекса

Лексический индекс лежит в отдельном SQLite-файле (`BSL_CODE_SEARCH_SQLITE_PATH`) на именованном
volume, отдельно от входных данных 1С. Векторные представления — в Neo4j. Для мультипроекта путь к
sidecar можно переопределять на каждый проект.

## Переиндексация

Полная переиндексация запускается автоматически, когда система обнаруживает несовместимость: изменился
отпечаток исходных данных, поменялась версия алгоритма нарезки/извлечения или явно запрошен reindex.
Версии алгоритмов повышаются вручную при изменении логики — это и служит триггером перестройки индекса.

## Типовые проблемы

- **`search_bsl_code` отсутствует среди инструментов** — не включён `ENABLE_BSL_CODE_SEARCH` или
  `LOAD_BSL_SIGNATURES` (нужны оба), либо инструмент скрыт в веб-консоли.
- **Поиск отвечает «not ready»** — индекс ещё строится. Индексация тел кода идёт после основной
  загрузки; дождитесь завершения.
- **Поиск находит по словам, но не по смыслу** — не включён `ENABLE_BSL_CODE_EMBEDDING` или ещё не
  прошла фаза B (эмбеддинги).
- **Нужные процедуры не находятся вообще** — проверьте, не попала ли их категория в
  `BSL_CODE_EMBEDDING_EXCLUDED_OWNER_CATEGORIES`.
