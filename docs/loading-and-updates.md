# Загрузка и обновление данных

Жизненный цикл данных: что и когда загружается в граф, как управлять составом загрузки флагами, как
работает инкрементальное обновление и когда нужна полная перезагрузка. Про запуск см.
[setup.md](setup.md), про модель графа — [architecture.md](architecture.md).

## Первичная загрузка

При первом старте контейнера приложение:

1. создаёт индексы в Neo4j (структурные и полнотекстовые, при необходимости — векторные);
2. загружает метаданные из выбранного источника (`METADATA_SOURCE=xml`, по умолчанию — XML-дамп
   напрямую, без текстового отчёта; `txt` — отчёт по конфигурации, доступен только при
   `PROJECT_LAYOUT=legacy`, см. [setup.md](setup.md));
3. при включённых флагах догружает из XML-выгрузки формы, предопределённые значения, права ролей,
   подписки на события, модули с процедурами и функциями, справку, расширения;
4. если включена векторная индексация — запускает её **в фоне** после загрузки основных данных.
   Во время векторной индексации MCP-сервер уже доступен для запросов.

Порядок по времени: сначала строится граф, затем (опционально) векторные эмбеддинги описаний и кода.

## Производительность и стоимость

Ориентир по времени и деньгам — замер на типовой конфигурации «Бухгалтерия КОРП» с настройками
загрузки по умолчанию (регламентированные отчёты в индексацию кода по умолчанию не попадают):

- **Загрузка графа** — ~30 минут (структура конфигурации: метаданные, формы, права, подписки, модули
  с процедурами/функциями и т.д.).
- **Фаза A индексации кода** (см. [bsl-code-search.md](bsl-code-search.md#как-устроен-индекс-две-фазы))
  — ~35 минут; параллельно с ней успевают пройти эмбеддинг метаданных, генерация сводок объектов и их
  эмбеддинг.
- **Фаза B — эмбеддинг тел кода** (провайдер по умолчанию — OpenRouter, модель
  `qwen/qwen3-embedding-4b`) — самая долгая часть: 4–6 часов, сильно зависит от загруженности
  провайдера. Продолжается в фоне уже после того, как MCP-сервер доступен для запросов.

Стоимость на той же конфигурации через OpenRouter:

- генерация сводок объектов (`gpt-5.4-mini`) — порядка **$16**;
- эмбеддинг (код + метаданные, `qwen3-embedding-4b`) — порядка **$3.5**.

Память:

- во время загрузки графа контейнер приложения потребляет порядка **7 ГБ**, контейнер Neo4j —
  порядка **5 ГБ**;
- после загрузки потребление приложения значительно снижается; у Neo4j — нет, кэш базы сбрасывается
  только перезапуском контейнера.

Цифры ориентировочные и зависят от объёма конфигурации, модели эмбеддингов и провайдера.

## Что грузится: флаги состава

Состав загрузки управляется флагами (значения и дефолты — в `.env.example`). Каждый флаг включает свою
часть графа:

| Флаг | Что добавляет в граф |
|------|----------------------|
| `LOAD_PREDEFINED_VALUES` | предопределённые значения (справочники, планы счетов и т.д.) |
| `LOAD_FORMS_FROM_XML` | формы: реквизиты, элементы, события, команды и привязка к обработчикам |
| `LOAD_ROLE_RIGHTS` | права ролей (связи доступа к объектам) |
| `LOAD_BSL_SIGNATURES` | сигнатуры процедур/функций из модулей и граф вызовов |
| `LOAD_HELP_FROM_HTML` | справку по объектам метаданных (для полнотекстового поиска по описаниям) |
| `LOAD_EVENT_SUBSCRIPTIONS` | подписки на события и их привязку к обработчикам |
| `LOAD_EXTENSIONS` | объекты расширений и их связи с базовой конфигурацией |
| `LOAD_METADATA_GUIDS` | обогащение узлов идентификаторами GUID из служебного дампа конфигурации |

Отключение флага означает, что соответствующая часть графа не строится, а зависящие от неё
инструменты MCP становятся недоступны (см. [mcp-tools.md](mcp-tools.md)). Загрузка тела кода BSL и
поиск по нему — отдельная подсистема, см. [bsl-code-search.md](bsl-code-search.md).

## Полная перезагрузка: `FULL_METADATA_RELOAD`

Полная перезагрузка очищает данные проекта в Neo4j и строит граф заново. Включается флагом
`FULL_METADATA_RELOAD=true` с последующим пересозданием контейнера:

```bash
# в docker-compose.yml для нужного сервиса
FULL_METADATA_RELOAD=true
docker compose up -d --force-recreate <имя-сервиса>
```

Полная перезагрузка обязательна, когда:

- изменился `METADATA_SOURCE` (`txt` ↔ `xml`) — базовые данные хранятся раздельно для двух источников,
  и инкрементальная синхронизация при несовпадении источника прерывается с ошибкой;
- нужно гарантированно пересобрать граф с нуля.

Флаг действует **одноразово на контейнер**: `true` выполняет full reload один раз для нового
контейнера, после чего обычный `docker restart` его не повторяет (даже если флаг остался `true`).
Одноразовость обеспечивается container-local marker-файлом в каталоге `/app/.container-state`, который
переживает `docker restart`, но исчезает при пересоздании контейнера. Поэтому:

- обычный рестарт безопасен и **не** перестраивает граф повторно;
- чтобы запустить full reload заново, пересоздайте контейнер (`--force-recreate`);
- если оставить `true`, любой будущий recreate снова выполнит full reload — при необходимости верните
  `false` после успешной загрузки.

## Инкрементальное обновление

Вместо полной перезагрузки система умеет догружать только изменения. Это управляется группой флагов
`INCREMENTAL_*` (см. секцию в `.env.example`).

- `INCREMENTAL_LOADING_ENABLED` — главный флаг подсистемы. Включает разовую проверку изменений после
  старта MCP и планировщик. При `false` поведение прежнее: обновление только через
  `FULL_METADATA_RELOAD` или первичную загрузку в пустую базу.
- `INCREMENTAL_LOADING_SCHEDULE_ENABLED` — включает периодические проверки по расписанию. При `false`
  выполняется только один проход при старте.
- `INCREMENTAL_LOADING_INTERVAL_MINUTES` — интервал между проверками планировщика.
- `INCREMENTAL_LOADING_OVERLAP_SECONDS` — запас по времени при отслеживании изменений файлов (защита
  от расхождения часов на диске).

Инкрементальная загрузка сравнивает текущие выгрузки с сохранённым состоянием и применяет только
дельту: добавленные, изменённые и удалённые объекты, формы, модули, права и т.д.

### Full reconcile

Инкрементальная дельта по watermark-ам не всегда замечает удаления. За это отвечает **full reconcile** —
полная сверка, которая находит удалённые объекты и артефакты (описатели метаданных, формы, модули,
предопределённые, справку — как в базовой конфигурации, так и в расширениях):

- `INCREMENTAL_FULL_RECONCILE_ENABLED` — включить сверку;
- `INCREMENTAL_FULL_RECONCILE_INTERVAL_HOURS` — периодичность;
- `INCREMENTAL_FULL_RECONCILE_WINDOW_START` / `_WINDOW_END` — окно локального времени, в которое
  разрешено запускать сверку (чтобы не мешать рабочей нагрузке).

## Хранение состояния

Прогресс инкрементальной загрузки хранится в отдельном SQLite-файле (`INCREMENTAL_LOADING_STATE_PATH`)
на именованном volume — отдельно от входных данных. Там же лежит блокировка планировщика, которая не
даёт двум циклам загрузки идти одновременно.

### Ограничение: один контейнер на состояние

Подсистема рассчитана на развёртывание, где **один контейнер приложения монопольно владеет своим
файлом состояния**. Флаг `INCREMENTAL_STARTUP_LOCK_TAKEOVER` по умолчанию позволяет процессу при
рестарте перехватить блокировку у предыдущего (упавшего) процесса — это корректно для одиночного
контейнера.

При развёртывании с несколькими репликами на **общем** состоянии этот перехват приведёт к тому, что
старт одного процесса отберёт блокировку у живого цикла другого и вызовет параллельную запись в
Neo4j. Для таких сценариев `INCREMENTAL_STARTUP_LOCK_TAKEOVER` нужно ставить в `false`, а лучше —
давать каждому процессу собственный файл состояния.

## Как читать логи загрузки

```bash
docker compose logs -f <имя-сервиса>
```

По логам видно: создание индексов, стадии загрузки метаданных и артефактов, старт и прогресс фоновой
векторной индексации, циклы инкрементального планировщика и reconcile.

## Типовые проблемы

- **Изменения в выгрузке не подхватываются** — проверьте, что включён `INCREMENTAL_LOADING_ENABLED`;
  удаления объектов замечает full reconcile, а не обычная дельта.
- **После смены `METADATA_SOURCE` инкремент падает с ошибкой** — это ожидаемо: поставьте
  `FULL_METADATA_RELOAD=true` и пересоздайте контейнер.
- **`FULL_METADATA_RELOAD=true` не сработал повторно** — это ожидаемо: флаг одноразовый на контейнер.
  Обычный `docker restart` full reload не повторяет; пересоздайте контейнер
  (`docker compose up -d --force-recreate <сервис>`).
- **`previous full reload attempt did not complete`** — предыдущая попытка full reload оборвалась
  (marker в статусе `started`/`failed`). Сервер намеренно не стартует, чтобы не обслуживать частично
  очищенный граф — **и при `FULL_METADATA_RELOAD=false` тоже**. Устраните причину и пересоздайте
  контейнер.
- **Параллельная запись/конфликты при нескольких репликах** — см. ограничение про
  `INCREMENTAL_STARTUP_LOCK_TAKEOVER` выше.
