# Установка и запуск

Как развернуть систему в Docker, подготовить данные и обслуживать её в дальнейшем. Для понимания
внутреннего устройства см. [architecture.md](architecture.md); за жизненным циклом загрузки —
[loading-and-updates.md](loading-and-updates.md).

## Требования

- Docker и Docker Compose.
- Свободные порты: `7474` и `7687` (Neo4j), `6001` (MCP-сервер и веб-консоль). Порт приложения меняется
  под каждый проект.
- Оперативная память из расчёта ~4 ГБ на Neo4j плюс ~6 ГБ на каждую одновременно загружаемую базу.
  После первичной загрузки потребление значительно снижается.

## Шаг 1. Файл `.env`

Есть два образца, скопируйте нужный в `.env`:

- **`.env.example.minimal`** — минимальный набор: только основные переменные, которых в большинстве
  случаев достаточно, чтобы управлять функциональностью. Рекомендуется для старта.
- **`.env.example`** — полный перечень всех переменных с дефолтами и комментариями, для тонкой
  настройки.

Минимум для старта:

- `NEO4J_PASSWORD` — обязательно, пароль базы.
- `PROJECT_NAME` — человекочитаемое имя проекта (становится именем узла Project в графе).

В `.env` держат значения, общие для всех контейнеров. Значения, различающиеся между проектами
(имя проекта, порт, пути сайдкаров), удобнее задавать прямо в `docker-compose.yml`.

## Шаг 2. Файл `docker-compose.yml`

Скопируйте `docker-compose.example.yml` в ту же папку, где лежит `.env`, и переименуйте в
`docker-compose.yml`.

- Задайте `PROJECT_NAME` в сервисе приложения (сам сервис можно назвать как угодно).
- Если проект один — второй сервис-пример можно удалить целиком.
- Если проектов больше — продублируйте секцию сервиса, задайте другой `PROJECT_NAME` и **другой порт
  на хосте** (меняется только первое число в паре `6001:6001`).

## Шаг 3. Данные

Раскладка каталогов внутри `data/prj1/` (смонтированного в `/app/data`) зависит от `PROJECT_LAYOUT`
(см. `.env.example`). По умолчанию — `vanessa`.

### `PROJECT_LAYOUT=vanessa` (по умолчанию) — без файла отчёта

Конвенция [vanessa-bootstrap](https://github.com/Diselkin/vanessa-bootstrap). `data/prj1/` — это сам
git-чекаут (результат `git clone`/`git pull`), примонтированный как есть, без промежуточного
копирования или перекладывания файлов:

```text
data/prj1/                       # = корень git-репозитория
  src/cf/                        # основная конфигурация — XML-выгрузка
  src/cfe/<ИмяРасширения>/       # расширения — плоско, Configuration.xml сразу внутри
```

Никакого текстового «Отчёта по конфигурации» здесь не требуется — метаданные парсятся напрямую из
XML-дампа (`METADATA_SOURCE=xml`, единственный источник, поддерживаемый этой раскладкой).

**GUID-обогащение (`LOAD_METADATA_GUIDS`) без файла в git.** `ConfigDumpInfo.xml` обычно держат вне
репозитория (`.gitignore`) — при групповой разработке нежелательно тащить постоянно меняющийся служебный
файл в историю. Механизм обогащения не завязан на git: приложение просто проверяет, лежит ли файл рядом
с XML-дампом на диске —

- `src/cf/ConfigDumpInfo.xml` — для базовой конфигурации;
- `src/cfe/<ИмяРасширения>/ConfigDumpInfo.xml` — для каждого расширения (у расширений тоже есть свой
  `ConfigDumpInfo.xml` при выгрузке).

Рабочая схема: та же автоматика, что выгружает конфигурацию в `.txt`/XML и коммитит `src/` в git (у неё
всё равно есть живое подключение к базе в момент выгрузки), кладёт `ConfigDumpInfo.xml` не в
основную историю, а в отдельную orphan-ветку (например, `guid-dump`), которую force-push’ит целиком на
каждом цикле — без накопления истории и без конфликтов при слиянии `main`/`dev`. Деплой-пайплайн после
`git clone`/`git pull` основного чекаута дополнительно накладывает файлы из этой ветки поверх уже
полученного `src/cf/` и `src/cfe/<Name>/` (`git pull` не трогает untracked/гитигнорённые файлы, так что
оверлей переживает последующие обновления чекаута). Никаких дополнительных настроек в приложении для
этого не нужно — оно просто увидит файлы на месте при следующей проверке.

**Автоматическое обновление по пушу в `main` (Gitea Actions).** Готовый пример workflow —
[`examples/gitea-actions/reindex-on-push.yml`](../examples/gitea-actions/reindex-on-push.yml). Кладётся
в `.gitea/workflows/` **репозитория конфигурации** (не этого репозитория), выполняется self-hosted
runner-ом на том же хосте, что и `docker compose`, и при пуше в `main`:

1. обновляет чекаут в каталоге, смонтированном как `/app/data`;
2. накладывает `ConfigDumpInfo.xml` из orphan-ветки `guid-dump` поверх `src/cf/` и `src/cfe/<Name>/`;
3. делает `docker restart <service>` (**не** `--force-recreate` и **не** `FULL_METADATA_RELOAD=true`) —
   это лишь заставляет пройти встроенный startup-инкремент немедленно вместо ожидания планового цикла
   (по умолчанию раз в час, `INCREMENTAL_LOADING_INTERVAL_MINUTES`). Данные не стираются, подхватывается
   только дельта — обычный `docker restart` не запускает full reload (см.
   [loading-and-updates.md](loading-and-updates.md#полная-перезагрузка-full_metadata_reload)).

Полный force-recreate нужен отдельно и редко — только при смене `METADATA_SOURCE`/`PROJECT_LAYOUT` или
для гарантированной пересборки графа с нуля, не как часть обычного цикла «запушили — переиндексировали».

### `PROJECT_LAYOUT=legacy`

Прежняя раскладка:

```text
data/prj1/
  metadata/     # выгрузка "Отчёт по конфигурации" (при METADATA_SOURCE=txt) — один .txt файл
  code/         # выгрузка "Выгрузить конфигурацию в файлы" (XML)
  extensions/   # (опционально) расширения — по папке на расширение, с той же
                # структурой metadata/+code/ внутри
```

Как получить выгрузки в Конфигураторе:

- **metadata** — `Конфигурация → Отчёт по конфигурации` (в текстовый файл, «Вся конфигурация»). В папке
  должен лежать ровно один `.txt`.
- **code** — `Конфигурация → Выгрузить конфигурацию в файлы` (XML).

Для остальных проектов структура аналогична (`data/prj2/...` и т.д.).

### Источник метаданных: `METADATA_SOURCE`

- `xml` (по умолчанию) — метаданные парсятся напрямую из XML-дампа (`src/cf/` в `vanessa`, `code/` в
  `legacy`);
- `txt` — метаданные берутся из `metadata/*.txt` (отчёт по конфигурации). Доступен только при
  `PROJECT_LAYOUT=legacy` — с `vanessa` приложение откажется стартовать с понятной ошибкой (в этой
  раскладке каталога `metadata/` попросту нет).

Смена источника на уже загруженном проекте не подхватывается инкрементально — нужен полный
перезапуск загрузки (см. [loading-and-updates.md](loading-and-updates.md)).

## Шаг 4. Запуск

```bash
docker compose up -d
```

При первом старте приложение создаст индексы в Neo4j и начнёт загрузку. Ход загрузки виден в логах:

```bash
docker compose logs -f <имя-сервиса>
```

## Сервисы после запуска

- **Neo4j Browser** — http://localhost:7474 (логин `neo4j`, пароль из `NEO4J_PASSWORD`).
- **Bolt** — `bolt://localhost:7687`.
- **MCP-сервер** — http://localhost:6001/mcp (порт зависит от проекта; путь настраивается `MCP_PATH`).
- **Веб-консоль** — http://localhost:6001/console (если `WEB_CONSOLE_ENABLED=true`).

Подключение MCP-клиентов описано в [mcp-tools.md](mcp-tools.md).

## Тома и служебное состояние

Входные данные монтируются из `data/`. Служебное состояние (состояние инкрементальной загрузки,
индекс поиска по коду, пользователи консоли) хранится в каталоге `storage/` на **именованных
volume-ах** — отдельно от `data/`. Сгенерированные сводки объектов лежат отдельно — файлами в
`data/object_summary/`. Практические следствия:

- пересоздание `data/` не сбрасывает прогресс загрузки, но удаляет сводки объектов;
- удаление volume-ов со `storage/` заставит систему переиндексировать всё заново;
- **у каждого проекта (сервиса) должен быть свой отдельный volume на `/app/storage`.** В примере
  compose это `storage_prj1`, `storage_prj2` и т.д. — по одному на сервис. Тогда служебные базы
  проектов (индекс поиска по коду, состояние загрузки) изолированы и не перезаписывают друг
  друга; переопределять пути внутри `storage/` не нужно.

> Отступление от этой схемы (один общий том на несколько проектов) требует вручную задать разные пути
> для sidecar-баз каждого проекта — см. переопределения `*_PRJ1`/`*_PRJ2` в `.env.example`. Штатный
> путь — отдельный том на проект.

## Обновление контейнеров

Обновление приложения на новую версию образа:

```bash
docker compose pull
docker compose up -d --force-recreate
```

Полный сброс с удалением базы (начать с нуля):

```bash
docker compose down --volumes
docker compose up -d --force-recreate
```
