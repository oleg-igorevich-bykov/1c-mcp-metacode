# Установка и запуск

Как развернуть систему в Docker, подготовить данные и обслуживать её в дальнейшем. Для понимания
внутреннего устройства см. [architecture.md](architecture.md); за жизненным циклом загрузки —
[loading-and-updates.md](loading-and-updates.md).

## Требования

- Docker и Docker Compose.
- Свободные порты: `7474` и `7687` (Neo4j), `6001` (MCP-сервер и веб-консоль). Порт приложения меняется
  под каждый проект.
- Оперативная память из расчёта ~4 ГБ на Neo4j плюс ~6 ГБ на каждую одновременно загружаемую базу.
  После первичной загрузки потребление значительно снижается.

## Шаг 1. Файл `.env`

Есть два образца, скопируйте нужный в `.env`:

- **`.env.example.minimal`** — минимальный набор: только основные переменные, которых в большинстве
  случаев достаточно, чтобы управлять функциональностью. Рекомендуется для старта.
- **`.env.example`** — полный перечень всех переменных с дефолтами и комментариями, для тонкой
  настройки.

Минимум для старта:

- `NEO4J_PASSWORD` — обязательно, пароль базы.
- `PROJECT_NAME` — человекочитаемое имя проекта (становится именем узла Project в графе).

В `.env` держат значения, общие для всех контейнеров. Значения, различающиеся между проектами
(имя проекта, порт, пути сайдкаров), удобнее задавать прямо в `docker-compose.yml`.

## Шаг 2. Файл `docker-compose.yml`

Скопируйте `docker-compose.example.yml` в ту же папку, где лежит `.env`, и переименуйте в
`docker-compose.yml`.

- Задайте `PROJECT_NAME` в сервисе приложения (сам сервис можно назвать как угодно).
- Если проект один — второй сервис-пример можно удалить целиком.
- Если проектов больше — продублируйте секцию сервиса, задайте другой `PROJECT_NAME` и **другой порт
  на хосте** (меняется только первое число в паре `6001:6001`).

## Шаг 3. Данные

В корне папки с `.env` и `docker-compose.yml` создайте структуру, монтируемую в контейнер. Для проекта
`prj1`:

```text
data/prj1/
  metadata/     # выгрузка "Отчёт по конфигурации" (при METADATA_SOURCE=txt) — один .txt файл
  code/         # выгрузка "Выгрузить конфигурацию в файлы" (XML)
  extensions/   # (опционально) расширения — по папке на расширение
```

Как получить выгрузки в Конфигураторе:

- **metadata** — `Конфигурация → Отчёт по конфигурации` (в текстовый файл, «Вся конфигурация»). В папке
  должен лежать ровно один `.txt`.
- **code** — `Конфигурация → Выгрузить конфигурацию в файлы` (XML).

Для остальных проектов структура аналогична (`data/prj2/...` и т.д.).

### Источник метаданных: `METADATA_SOURCE`

- `txt` — метаданные берутся из `metadata/*.txt` (отчёт по конфигурации);
- `xml` — метаданные парсятся напрямую из XML-дампа в `code/`.

Смена источника на уже загруженном проекте не подхватывается инкрементально — нужен полный
перезапуск загрузки (см. [loading-and-updates.md](loading-and-updates.md)).

## Шаг 4. Запуск

```bash
docker compose up -d
```

При первом старте приложение создаст индексы в Neo4j и начнёт загрузку. Ход загрузки виден в логах:

```bash
docker compose logs -f <имя-сервиса>
```

## Сервисы после запуска

- **Neo4j Browser** — http://localhost:7474 (логин `neo4j`, пароль из `NEO4J_PASSWORD`).
- **Bolt** — `bolt://localhost:7687`.
- **MCP-сервер** — http://localhost:6001/mcp (порт зависит от проекта; путь настраивается `MCP_PATH`).
- **Веб-консоль** — http://localhost:6001/console (если `WEB_CONSOLE_ENABLED=true`).

Подключение MCP-клиентов описано в [mcp-tools.md](mcp-tools.md).

## Тома и служебное состояние

Входные данные монтируются из `data/`. Служебное состояние (состояние инкрементальной загрузки,
индекс поиска по коду, пользователи консоли) хранится в каталоге `storage/` на **именованных
volume-ах** — отдельно от `data/`. Сгенерированные сводки объектов лежат отдельно — файлами в
`data/object_summary/`. Практические следствия:

- пересоздание `data/` не сбрасывает прогресс загрузки, но удаляет сводки объектов;
- удаление volume-ов со `storage/` заставит систему переиндексировать всё заново;
- **у каждого проекта (сервиса) должен быть свой отдельный volume на `/app/storage`.** В примере
  compose это `storage_prj1`, `storage_prj2` и т.д. — по одному на сервис. Тогда служебные базы
  проектов (индекс поиска по коду, состояние загрузки) изолированы и не перезаписывают друг
  друга; переопределять пути внутри `storage/` не нужно.

> Отступление от этой схемы (один общий том на несколько проектов) требует вручную задать разные пути
> для sidecar-баз каждого проекта — см. переопределения `*_PRJ1`/`*_PRJ2` в `.env.example`. Штатный
> путь — отдельный том на проект.

## Обновление контейнеров

Обновление приложения на новую версию образа:

```bash
docker compose pull
docker compose up -d --force-recreate
```

Полный сброс с удалением базы (начать с нуля):

```bash
docker compose down --volumes
docker compose up -d --force-recreate
```
