# Задание: детальный прогресс bootstrap-индексации графа

Развитие фазы 2 из `TASK-index-status.md` (там появился анонимный
`GET /api/console/health/index`: только `{"ready": true|false}`). Сейчас
нужна детализация: какая фаза идёт, сколько обработано, сколько осталось.

## Контекст

Первичная (bootstrap) индексация графа конфигурации в 1c-mcp-metacode стартует
сама при первом запуске контейнера и может идти долго на большой
конфигурации. Снаружи сейчас видно только «готово / не готово» — не видно,
завис процесс или просто идёт медленная фаза, и сколько ждать.

Мониторинг (job `fleet-index-progress` в Prometheus) уже настроен и ждёт
метрики — см. `monitoring/README.md`, раздел «Прогресс индексации». Осталось
только реализовать эндпоинт в приложении.

## Что сделать

Добавить второй анонимный эндпоинт (существующий `/api/console/health/index`
не трогать — не меняем поведение уже подключённого проб-эндпоинта):

```
GET /api/console/metrics/index
```

На том же порту, что и MCP/консоль (6001), без авторизации, без данных
конфигурации — только числовые метрики о ходе индексации. Ответ — **не
JSON**, а текст в формате Prometheus exposition (`Content-Type:
text/plain; version=0.0.4`), чтобы Prometheus мог его скрейпить напрямую.

### Метрики (контракт)

```
# TYPE mcp_index_ready gauge
mcp_index_ready 0
# TYPE mcp_index_progress_ratio gauge
mcp_index_progress_ratio 0.55
# TYPE mcp_index_phase_info gauge
mcp_index_phase_info{phase="building_graph"} 1
# TYPE mcp_index_processed_items gauge
mcp_index_processed_items 6789
# TYPE mcp_index_total_items gauge
mcp_index_total_items 12345
# TYPE mcp_index_eta_seconds gauge
mcp_index_eta_seconds 320
```

Правила:

- `mcp_index_ready` — 1:1 с булевым полем `ready` из `/api/console/health/index`
  (0/1), чтобы прогресс и готовность были видны в одном запросе Prometheus.
- `mcp_index_progress_ratio` — 0..1. Когда `ready=1`, отдавайте 1 (или просто
  не публикуйте метрику — Grafana покажет её как «нет данных», это нормально
  для завершённого индекса, если вам проще не считать прогресс после финиша).
- `mcp_index_phase_info{phase="..."}` — метрика-индикатор (info-паттерн):
  всегда ровно ОДНА активная фаза со значением `1`, остальные фазы в ответе
  не публикуются (не 0 — просто отсутствуют). Имена фаз — на ваше усмотрение,
  важно только чтобы они были стабильными строками (не менялись между
  версиями без причины), например: `loading_metadata`, `building_graph`,
  `embedding_code`, `generating_summaries`, `incremental_update`. Список для
  справки — Grafana просто выводит их как есть в таблице, семантика вам
  известна лучше.
- `mcp_index_processed_items` / `mcp_index_total_items` — единица измерения
  (объекты метаданных, узлы графа, файлы — что угодно) на ваше усмотрение,
  главное чтобы processed/total были в ОДНОЙ и той же единице (иначе
  `progress_ratio` и % разъедутся). Если для конкретной фазы общее число
  заранее неизвестно — `total_items` не публикуйте вовсе (не 0 и не -1;
  отсутствие метрики Prometheus обработает корректно, деление на ноль не
  создаст).
- `mcp_index_eta_seconds` — оценка оставшегося времени. Если оценить нельзя
  (в начале фазы, скорость ещё не устаканилась) — не публикуйте метрику,
  не подставляйте условный 0 или -1: любое число будет показано в дашборде
  как факт, а не как «неизвестно».
- Никаких меток вроде `fleet_unit`/`repo`/`branch` добавлять не нужно —
  Prometheus проставит их сам из имени контейнера (file_sd), достаточно
  `phase` на `mcp_index_phase_info`.

### Требования к эндпоинту

- Анонимный (как `/api/console/health/index`), без токена.
- Не должен создавать заметную нагрузку — Prometheus дёргает его раз в 30
  секунд на каждый живой graph-контейнер; значения должны браться из уже
  поддерживаемого в памяти состояния индексатора, не пересчитываться с нуля.
- Пока индексация не началась или уже завершена — эндпоинт всё равно должен
  отвечать 200 (это метрики о процессе, а не readiness-проверка); публикуйте
  минимум `mcp_index_ready`.

## Критерии приёмки

1. `curl -s http://<graph-контейнер>:6001/api/console/metrics/index` без
   заголовков авторизации отдаёт 200 и текст в Prometheus exposition
   формате (парсится `promtool check metrics` или `prometheus_client`
   парсером на стороне проверяющего).
2. Во время реальной bootstrap-индексации: `mcp_index_ready=0`,
   `mcp_index_phase_info` показывает ровно одну активную фазу,
   `mcp_index_processed_items` монотонно растёт между последовательными
   запросами.
3. После завершения индексации: `mcp_index_ready=1`,
   `/api/console/health/index` тоже отдаёт `ready: true` (согласованность
   между двумя эндпоинтами).
4. `mcp_index_total_items` не публикуется (либо публикуется корректно), если
   реальное общее число неизвестно на момент запроса — не 0.
5. Нагрузка от опроса раз в 30с не создаёт заметной просадки на самом
   graph-сервисе (используются уже посчитанные внутренние счётчики).
